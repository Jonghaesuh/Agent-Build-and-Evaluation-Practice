# Data & Model Agent

PRD → TASK 구체화 → 데이터 전처리 → 후보 모델 개발 → 검증 코드 → Fallback 코드까지의
전체 파이프라인을 다룬다. 이 skill은 **하나의 긴 실행 흐름이 아니라 6단계로 나뉘며,
3개의 Human Checkpoint에서 반드시 멈춰서 확인을 받아야 한다.**

## 언제 사용하는가
- PRD 문서(신규/수정)가 주어졌을 때
- 가용 데이터를 바탕으로 TASK를 다시 정리해야 할 때
- 전처리 로직/코드, 후보 모델 코드, 튜닝 코드, Fallback 코드 중 하나라도 필요할 때

## 전체 흐름

```
PRD 입력
  → [1] TASK 구체화 ────────────────► Checkpoint #1 (TASK 확정)
  → [2] 데이터 프로파일링 + 전처리 설계 ─► Checkpoint #2 (전처리 로직 확정)
  → [3] 전처리 코드 구현
  → [4] 후보 모델(2~3개) 탐색 + 코드 작성 ► Checkpoint #3 (후보 모델 선정)
  → [5] train/val/test + 튜닝 코드 작성
  → [6] Fallback/오류 처리 코드 작성
```

각 Checkpoint에서는 결과를 요약해 사용자에게 제시하고, 명시적 승인(또는 수정 지시)을
받은 뒤에만 다음 단계로 진행한다. 승인 없이 임의로 다음 단계 코드를 먼저 작성하지 않는다.

---

## [1] TASK 구체화

1. PRD를 읽고 고객 요구사항/기능을 목록화한다.
2. 각 TASK 항목에 대해:
   - PRD의 어느 부분(섹션/문장 위치)에서 도출됐는지 **paraphrase 요약 + 위치 참조**로 근거를 남긴다. PRD 원문을 그대로 길게 인용하지 않는다.
   - 우선순위 태그를 붙인다: `Must` / `Should` / `Could`. (최종 확정은 사용자 몫)
3. PRD에 명시되지 않았지만 암묵적으로 필요해 보이는 요구사항(응답 속도, 예외 기준 등)은
   별도로 "확인 필요 항목" 리스트로 정리한다.
4. TASK 목록 + 확인 필요 항목을 사용자에게 제시하고 **Checkpoint #1**로 승인을 요청한다.
   승인 요청 시 "승인하시면 다음으로 가용 데이터 프로파일링과 전처리 설계안을
   진행합니다"와 같이 다음 단계를 한 줄로 미리 안내한다.

## [2] 데이터 파악 및 전처리 설계

1. 가용 데이터를 LOAD하기 전에 프로파일링부터 수행한다:
   - 스키마/컬럼 타입, 결측치 비율, 이상치 분포, 클래스 불균형 여부
2. 프로파일링 결과를 바탕으로 TASK 목록을 필요 시 업데이트한다(데이터로 실현 불가능한
   TASK가 있으면 명시적으로 알린다).
3. 전처리 설계안을 **코드 작성 전에 먼저 문서로 제시**한다: 결측치 처리 방식, 정규화/
   인코딩 방식, 샘플링 전략 등.
4. 설계안을 사용자에게 제시하고 **Checkpoint #2**로 승인을 요청한다. 승인 전에는
   전처리 코드를 작성하지 않는다. 승인 시 다음 단계(전처리 코드 구현 및 실행 검증)를
   한 줄로 미리 안내한다.

## [3] 전처리 코드 구현

- 승인된 설계안대로 구현한다. 원본 데이터는 절대 직접 변형하지 않고, 재현 가능한
  함수/파이프라인 객체(예: `sklearn.Pipeline` 또는 자체 클래스)로 작성한다.
- 코드는 sandbox에서 최소 1회 실행해 성공/실패 로그와 샘플 출력(전처리 전후 비교)을
  함께 제시한다.

## [4] 후보 모델 탐색 및 개발

1. TASK 성격(분류/회귀/시계열/텍스트 등)에 맞는 후보 모델을 **2~3개** 선정한다.
2. 각 후보에 대해 비교표를 작성한다:

   | 후보 | 연산 비용(학습/추론) | 해석 가능성 | 데이터 규모 적합성 | 예상 리스크 |
   |---|---|---|---|---|

3. 각 후보는 독립 실행 가능한 코드 단위로 작성하고, 공통 인터페이스
   (`fit`, `predict`, `evaluate` 등)를 통일한다.
4. 비교표와 코드를 제시하고 **Checkpoint #3**으로 최종 모델 선정을 요청한다. 승인 시
   다음 단계(train/val/test 및 튜닝 코드, Fallback 코드 작성)를 한 줄로 미리 안내한다.

## [5] train/val/test 및 튜닝 코드

- 분할 비율과 방식(랜덤/시계열/계층적)을 정하고 **선택 근거**를 함께 기술한다.
- 튜닝 코드는 탐색 범위(search space)와 탐색 방법(Grid/Random/Bayesian 등)을
  코드 주석 또는 문서에 명시한다.

## [6] Fallback / 오류 처리 코드

다음 케이스별 기본 동작을 빠짐없이 정의하고 코드화한다:

| 케이스 | 기본 동작 예시 |
|---|---|
| 데이터 결측/스키마 불일치 | 기본값 대체 / 스킵 / 파이프라인 중단 — 택1 + 사유 명시 |
| 이상치/범위 초과 입력 | clipping, 로깅 후 제외 |
| 모델 추론 실패/타임아웃 | retry 횟수 지정 + fallback(규칙 기반 기본값 등) |
| 신규/미지의 카테고리 값 | unknown 카테고리 처리 로직 |

모든 예외는 발생 시각/원인/처리 방식을 포함한 로그로 남긴다.

## 산출물 체크리스트

- [ ] 데이터 분석 플로우 요약 문서
- [ ] 데이터 전처리 코드 (+ 실행 검증 로그)
- [ ] ML/DL 모델 개발 코드 — 후보 2~3개 (+ 실행 검증 로그)
- [ ] train/val/test 데이터 처리 및 튜닝 코드
- [ ] Fallback/오류 처리 코드

## 지켜야 할 규칙 (요약)

- 모든 산출물은 사람 승인 전까지 "draft"로 표시한다.
- PRD 원문을 임의로 수정/추가/축소하지 않는다. 모호하면 가정을 명시하고 확인을 요청한다.
- 모든 판단에는 근거를 남긴다(추적 가능성).
- 코드는 실행 검증 없이 최종 산출물로 제시하지 않는다.
- 고객 데이터를 승인되지 않은 외부 서비스로 전송하지 않는다.
- 성능/품질을 과장된 표현으로 서술하지 않는다.
- 3개의 Human Checkpoint(TASK 확정, 전처리 로직 확정, 후보 모델 선정)를 임의로
  건너뛰지 않는다.
